## 2. Purpose / Big Picture

**Rollout note (read this first).** This agent build is gated on a corresponding backend change. Today's backend sends relative filepaths like `v1/motion-control.json`. After this change the agent rejects every non-absolute filepath as a backend bug. Do NOT deploy this agent build to a fleet whose backend has not been refactored to emit absolute filepaths.

The Miru agent deploys config instances by writing files to disk on customer devices. Today's code joins the backend-supplied `filepath` under a hardcoded managed root (`/srv/miru/config_instances/`). After this change the agent treats `cfg_inst.filepath` as the literal absolute path it should write to — no joining, no normalization, no rerouting. Customers can deliver config files anywhere on the device's filesystem, gated only by standard Unix permissions on the `miru` user/group.

User-visible behavior after the change:

1. The operator decides where they want config files to land (e.g. `/etc/myapp/`) and grants the `miru` user write access using standard Unix tools:

       sudo install -d -g miru -m 775 /etc/myapp

2. The backend sends a config instance with `filepath = "/etc/myapp/config.json"`. The agent transactionally copies the cached source content to that literal path and reports `Deployed`. Multi-file deployments either commit all destinations or roll every destination back to its pre-deployment state (per D3).

3. If the backend sends a non-absolute filepath, the agent emits `PathNotAllowed { reason: "filepath is not absolute" }`. The deploy state machine logs the error and schedules a retry per the existing `RetryPolicy` (per D4) — once the operator fixes the upstream backend, the next retry succeeds.

4. If the destination's parent directory is not writable by the `miru` user, the kernel returns `EACCES`. The agent translates this to `PathNotAllowed { reason: "permission denied; ..." }`, which the operator sees on every retry. The operator runs `sudo chgrp miru <dir> && sudo chmod g+w <dir>` and the next retry succeeds.

5. Customers who want stricter confinement can ship the example lockdown drop-in (D2 and Plan of Work part G), which re-enables systemd bind-mount sandboxing on top of Unix DAC.

## 4. Surprises & Discoveries

- **No existing `DeployErr → DplErrStatus` match to add an arm to.** The plan
  for Part H assumed a match statement already maps `DeployErr` variants to
  `DplErrStatus`. The actual wiring today flows through `fsm::error`, which
  only distinguishes errors via the generic `Error::is_network_conn_err()`
  hook — every non-network error is funneled through the retry path and can
  only reach `Failed` once `attempts >= max_attempts`. To honour the plan's
  intent that `PathNotAllowed` short-circuits straight to `Failed`, I added a
  new `fsm::fail()` terminal transition (patches `error_status = Failed` with
  `cooldown = 0`) and a `transition_on_error()` helper in `apply.rs` that
  matches on the `DeployErr` variant and routes `PathNotAllowed` to
  `fsm::fail`, everything else to `fsm::error`. Non-`PathNotAllowed` errors
  keep their existing retry behaviour.
- **`Layout::customer_configs()` is still live.** The plan noted that
  `Layout::srv_temp_dir()` might become dead and should get a TODO comment.
  I confirmed this and added the TODO. `customer_configs()` is still called
  from `storage::setup::bootstrap` (wipes the legacy config instances
  directory during device bootstrap), so it is NOT dead yet and was left
  untouched.
- **`write_file()` is now unreferenced.** Part E of the plan rewrites
  `write_file()` as a thin transactional wrapper, and Part F rewrites
  `write_files()` to build the writes vec inline rather than calling
  `write_file()`. That leaves `write_file()` without a caller. Kept it in
  the tree (gated with `#[allow(dead_code)]`) per the plan's explicit
  "rewrite, don't delete" wording; a future cleanup pass can drop it.
- **`atomicwrites::Error` is already converted to `io::Error` inside
  `File::write_bytes()`.** The plan's chain description
  (`FileSysErr::AtomicWriteFileErr → atomicwrites::Error → std::io::Error`)
  is really one hop: `write_bytes` does `.map_err(|e| e.into())` before
  storing the error in `AtomicWriteFileErr::source`. `translate_transactional_err`
  handles this by walking the generic `Error::source()` chain and returning
  the innermost `io::Error::kind()` it finds, which works regardless of
  whether the chain has one hop or several.
- **`innermost_io_kind` silently fell through for `Box<io::Error>` sources.**
  Writing the integration tests for the EACCES-rollback path exposed a real
  bug: `FileSysErr` variants store their source as `Box<std::io::Error>`, and
  thiserror's derived `source()` method routes boxed sources through
  `Box<T>::as_dyn_error()`. That returns a `&dyn Error` whose concrete type
  is `Box<io::Error>`, not `io::Error`, so the original
  `e.downcast_ref::<std::io::Error>()` call always returned `None` and the
  walker fell through to the `_ => DeployErr::FileSysErr(source)` arm even
  for permission-denied writes. Fixed by adding a `Box<std::io::Error>`
  downcast fallback inside the walk loop. Without this fix every integration
  test that expects `DeployErr::PathNotAllowed` for a real EACCES write would
  have produced `DeployErr::FileSysErr` and a misrouted retry.
- **Sync helper `make_cfg_inst` needed an absolute filepath.** Twenty
  pre-existing sync tests used `format!("{id}.json")` as the config instance
  filepath, which the new `validate_filepath` check rejects as "not
  absolute". Rewrote the helper to build the filepath under a per-process
  `/tmp/miru-sync-test-{pid}/` directory so parallel / repeated runs don't
  collide. The sync tests then exercise the full pull/apply/push flow
  without tripping validation.

### Second-pass revision (2026-04-07)

- **Content cache stored JSON-wrapped `CacheEntry`, not raw bytes.** The
  plan assumed the existing `CfgInstContent` cache stored content as raw
  bytes on disk so the deploy layer could `tokio::fs::copy` from the cache
  file directly. Investigation showed `CfgInstContent` was a type alias
  for `cache::DirCache<CfgInstID, String>`, and `DirCache` serializes
  every entry as a `CacheEntry { key, value, dirty, ... }` JSON wrapper.
  Passing those bytes as a copy source would silently corrupt the
  customer's destination file with the JSON wrapper. To honor the plan's
  intent (no in-memory hop), I replaced the type alias with a custom
  `CfgInstContent` struct that stores raw bytes on disk. The public API
  is preserved verbatim so existing call sites (sync, deploy, tests)
  work unchanged. The struct uses a `.bin` extension instead of the old
  `.json` so an in-place agent upgrade ignores old DirCache-format files
  that would otherwise be served as raw content.
- **The new content store had no LRU eviction (regression-free).** The
  initial second-pass implementation introduced an LRU `prune` step that
  ran on every `write` and could evict files mid-deploy, creating a
  TOCTOU race with the deploy layer's `path()` accessor. The original
  `DirCache` had no automatic eviction either (only `prune_invalid_entries`
  for un-deserializable files), so removing the new `prune` restores
  parity with the previous behavior and eliminates the race entirely.
  Documented in the struct's docstring.
- **`tokio::fs::copy` and `tokio::fs::rename` do not auto-create parent
  directories.** The previous transactional helper went through
  `File::write_bytes`, which called `self.parent()?.create_if_absent()`
  internally. The new filesystem-only helper bypasses that, so any
  destination whose parent directory did not already exist would fail
  with `ENOENT`. Fixed by adding an explicit `tokio::fs::create_dir_all`
  step at the top of each iteration in `write_all`, mapped to
  `SnapshotFailed` on failure. Two existing integration tests
  (`nested_parent_dirs_are_created`, `nested_filepath`) caught this
  during the test analysis pass.
- **`cleanup_backups` was being called on the `RolledBack` path.** The
  initial second-pass implementation called `cleanup_backups` before
  every `RolledBack` return, which was a no-op (the rollback's atomic
  rename had already consumed each `Existed` snapshot's backup) but
  muddied the invariant "RolledBack means rollback owns the backups".
  Removed in the refine pass. The plan's contract is that
  `cleanup_backups` runs on success and on `SnapshotFailed`, not on
  `RolledBack` and not on `RollbackIncomplete`.
- **`tokio::fs::copy` can leak partial files on mid-stream failures.**
  The initial implementation didn't clean up the destination of a failed
  `tokio::fs::copy` call, so a `dst.miru-tmp-<uuid>` or
  `dst.miru-backup-<uuid>` could be left behind on every failed deploy
  attempt. Added best-effort `remove_file` cleanup in both the snapshot
  copy and the src→tmp copy failure branches. Failures during cleanup
  are logged at warn level.
- **Second-pass storage rewrite was over-engineered.** Investigation
  during PR #22 review surfaced that the `CfgInstContent` rewrite (custom
  struct, raw bytes on disk, `.bin` extension, no LRU eviction, 13
  dedicated unit tests) was a much heavier change than the problem
  required. The actual constraint — "the source bytes passed to
  `tokio::fs::copy` must be exactly the destination's bytes" — can be
  satisfied by staging cached content to a per-call temp file in the
  deploy layer, with no changes to the storage module. The third-pass
  revision (D11) reverts the storage rewrite and adds the temp-staging
  loop.

## 5. Decision Log

### D1. Agent treats `filepath` as a literal absolute path

The agent does no path interpretation: no joining with a managed root, no normalization, no leading-slash heuristic, no `clean()`. `cfg_inst.filepath` is passed verbatim to `PathBuf::from()`. The backend is responsible for emitting the correct absolute filepath.

The agent applies two cheap inline sanity validations on the way in:

1. Reject if `Path::is_absolute()` is false → `PathNotAllowed { reason: "filepath is not absolute" }`.
2. Reject if any component is `Component::ParentDir` → `PathNotAllowed { reason: "filepath contains parent traversal" }`.

These are rejections, not conversions. They surface malformed input upstream of the kernel with a clear operator-facing message.

### D2. Filesystem access is enforced by Unix DAC; bind-mount confinement is opt-in

The default systemd unit at `agent/build/debian/miru.service` does NOT enable `ProtectSystem=strict` or `ProtectHome=true`. Filesystem write access is enforced by standard Unix DAC permissions on the `miru` user/group. Customers grant the agent write access to a new directory by `chgrp miru <dir> && chmod g+w <dir>` (or `install -d -g miru -m 775 <dir>`). All other hardening directives stay (`User=miru`, `Group=miru`, `NoNewPrivileges=true`, `PrivateTmp=true`, `PrivateDevices=true`, `ProtectProc=invisible`, `RestrictAddressFamilies=...`).

`ReadWritePaths=/var/lib/miru /var/log/miru /srv/miru` is kept in the default unit even though it is a no-op without `ProtectSystem=strict`. systemd merges `ReadWritePaths=` additively across drop-ins, so any customer who later ships a lockdown drop-in with `ProtectSystem=strict` automatically gets the agent's own working directories pre-listed and only has to add their own.

A `lockdown.conf.example` file ships in the repo for customers who want bind-mount confinement on top of Unix DAC. They copy it to `/etc/systemd/system/miru.service.d/lockdown.conf`, edit `ReadWritePaths=`, then `daemon-reload` and restart `miru.service`. The agent does nothing with the drop-in at runtime — it's pure operator workflow.

Rationale: Unix DAC is the standard mechanism every operator already understands, lower friction than maintaining a parallel agent-side allowlist or forcing customers through systemd drop-ins. Robotics customers in particular often write config files under home directories, which `ProtectHome=true` would forbid even if Unix permissions allow. Defense in depth via the lockdown drop-in remains available for security-focused customers.

### D3. Best-effort transactional multi-file writes via filesystem operations

Today's `write_files()` stages every file in `/srv/miru/.temp/<uuid>/` and atomically renames the temp dir over `/srv/miru/config_instances/`. With absolute filepaths going to disjoint locations, that single-rename strategy is impossible. We replace it with a new transactional helper that gives best-effort all-or-nothing semantics for the END state of a deployment, using filesystem operations only — no in-memory buffering of file contents and no memory-resident snapshots.

The helper lives at `agent/agent/src/filesys/transactional.rs` (new module). It takes `&[(File, File)]` pairs where the first `File` is the **source** (a file already on disk that should remain untouched) and the second is the **destination**. For each `(src, dst)` pair, in order, it:

1. **Snapshots** the destination by checking whether it exists. If yes, copy the destination to a sibling backup file in the same parent directory (e.g. `<dst>.miru-backup-<uuid>`) using `tokio::fs::copy`, and record `Existed { dst, backup }`. If no, record `DidNotExist { dst }`.
2. **Writes** by atomically copying `src` to `dst`. The copy uses a sibling temp file (`<dst>.miru-tmp-<uuid>`) followed by `tokio::fs::rename` over `dst` — this is the same atomicity pattern `atomicwrites` uses internally, but applied to a streaming `tokio::fs::copy` rather than an in-memory buffer.
3. **On any failure** during either phase: stop the forward pass and walk back through completed snapshots in reverse, restoring each one — `Existed { dst, backup }` is restored via `tokio::fs::rename(backup, dst)`; `DidNotExist { dst }` is restored via `tokio::fs::remove_file(dst)`.
4. **On success**, walk the snapshots once more to remove every `Existed { backup }` file. Backup-cleanup failures are logged but do not fail the call — the backup files are harmless leftover bytes if they survive.

Three structured error categories tell the deploy layer how bad the situation is:

- `SnapshotFailed` — failure occurred before any destination writes were attempted (the snapshot copy itself failed for a still-existing destination, or the existence check failed). On-disk state is unchanged from the helper's perspective; orphan backup files from earlier successful snapshots in the same call are cleaned up before returning.
- `RolledBack` — a write failed and the rollback succeeded fully. On-disk state matches what it was before the call.
- `RollbackIncomplete` — a write failed AND one or more rollback restores also failed. On-disk state is partially updated. This is the worst case and the only one where the agent's view of the filesystem is genuinely inconsistent. Backup files are intentionally NOT cleaned up in this case so an operator can recover them manually.

Trade-offs and accepted limits:

- **No in-memory buffering.** Unlike a `(File, &[u8])` design that holds every file's new bytes in RAM until the call commits, this design streams source-to-destination via `tokio::fs::copy` and snapshot-to-backup via the same primitive. Memory usage is bounded by the per-syscall buffer the kernel uses, not by the total deployment size.
- **Source files are never modified.** The helper's contract is that every source `File` exists on disk before the call and exists with identical bytes after the call, regardless of success or failure. The helper does not touch the source under any code path.

Staging note: as of the third-pass revision (D11), the source files passed to this helper are per-call temp files staged by the deploy layer rather than long-lived cache files. The "source files are never modified" contract is still honored end-to-end: the helper does not touch the source, and the deploy layer is responsible for cleaning the staging temp files up after the call (on both success and error paths).
- **Sibling backup placement.** Backup and temp files live in the same parent directory as the destination, so they share a filesystem with `dst` (a requirement for the rename to be atomic). The downside is that the parent directory must be writable by the `miru` user — but that is exactly the access the agent needs anyway to write `dst` itself, so there is no additional permission requirement.
- **Concurrent third-party modifications.** The helper assumes the agent is the only writer to the destination paths. If a human or another process modifies a file mid-deployment, the rollback may clobber their changes. Same assumption every Miru-managed config file already makes.
- **Empty directories from `create_if_absent`.** Parent directories created during the destination write are not removed on rollback. They are harmless and the next deployment will reuse them.
- **Stale-file removal still out of scope.** Files from a previous deployment that are not in the current one are not deleted. A follow-up plan can add a cleanup pass driven by the deploy state machine.

### D4. All errors flow through the existing retry path

`DeployErr::PathNotAllowed` (and every other error from the deploy write path) is treated by the deploy state machine the same way today's filesystem errors are treated: it routes through `fsm::error`, which logs the failure and schedules a retry per the existing `RetryPolicy`. There is no terminal "hard fail" branch that short-circuits to `DplErrStatus::Failed` for any specific variant. The rationale: even configuration-mismatch errors like a missing `chgrp` or a backend that's still sending relative filepaths can become correct without redeploying the agent — once an operator fixes the situation, the next retry succeeds and the deployment moves to `Deployed`. Forcing a hard `Failed` would require operator intervention to re-trigger the deployment instead.

The trade-off is that genuinely-misconfigured deployments will retry forever (or until the existing `RetryPolicy::max_attempts` cap kicks in, which still ends in `Failed` after several attempts) instead of failing immediately. Operators who watch the agent logs will see clear `PathNotAllowed { reason: ... }` messages on every retry, which is the intended signal — the verbose logging is the point. A future iteration can add per-variant retry policy or a faster fail path if the always-retry behavior turns out to be too patient in practice.

### D5. Do not touch the OpenAPI spec or the generated client

The OpenAPI spec at `agent/api/specs/backend/v02.yaml` documents `filepath` as relative to `/srv/miru/config_instances/`. This plan deliberately does not update the spec or regenerate `agent/libs/backend-api/`. The reinterpretation is entirely agent-side. The spec update is part of the backend refactor (D6) and is out of scope here.

### D6. Backend dependency

## 7. Context and Orientation

The Miru agent is a Rust binary that runs on customer devices. It pulls deployments from the Miru backend, downloads config instance content, and writes files to local disk. Hot files for this plan:

- `agent/agent/src/deploy/filesys.rs` — `deploy()` (entry point) calls `validate_deployment` → `read_cfg_insts` → `validate_cfg_insts` → `write_files`. ### D. Transactional write helper (filesystem-only, no in-memory bytes)

Create a new module `agent/agent/src/filesys/transactional.rs` (and add `pub mod transactional;` to `agent/agent/src/filesys/mod.rs`).

Public surface:

    pub async fn write_all(writes: &[(File, File)]) -> Result<(), TransactionalWriteErr>

    pub enum TransactionalWriteErr {
        /// Snapshot copy failed before any destination writes were attempted.
        /// On-disk state is unchanged from the helper's perspective; orphan
        /// backup files from earlier successful snapshots in the same call
        /// have been cleaned up before the helper returns.
        SnapshotFailed { source: FileSysErr, file: File },
        /// A destination write failed and rollback restored every prior
        /// destination to its pre-call state.
        RolledBack { source: FileSysErr, failed_file: File },
        /// A destination write failed AND one or more rollback restores also
        /// failed. On-disk state is partially updated. Backup files for the
        /// affected destinations are intentionally NOT cleaned up so the
        /// operator can recover them manually.
        RollbackIncomplete {
            source: FileSysErr,
            failed_file: File,
            rollback_errors: Vec<FileSysErr>,
        },
    }

The `(File, File)` tuple is `(source, destination)`. The source must already exist on disk before the call. The contract is that the source is **never modified** by the helper — every read of the source is a copy operation; the source's bytes and inode are unchanged on success and on failure.

`TransactionalWriteErr` derives `thiserror::Error` and implements `crate::errors::Error` per the repo's existing pattern.

Internal snapshot type, scoped to the module:

    enum Snapshot {
        DidNotExist { dst: File },
        Existed { dst: File, backup: File },
    }

Internal helper for picking a sibling temp/backup filename, scoped to the module:

    fn sibling(dst: &File, suffix: &str) -> File {
        // <dst_parent>/.<dst_name>.miru-<suffix>-<uuid>
        // Use a leading dot so the temp/backup name is hidden in default
        // directory listings, and a uuid suffix to avoid collisions across
        // concurrent calls or per-deployment retries.
    }

Algorithm for `write_all`:

1. Initialize `let mut completed: Vec<Snapshot> = Vec::with_capacity(writes.len());`.
2. For each `(src, dst)` in `writes`, in order:
   - **Snapshot:** check `tokio::fs::try_exists(dst.path()).await`. If the existence check returns `Err`, run `cleanup_backups(&completed)` (best-effort) and return `TransactionalWriteErr::SnapshotFailed { source, file: dst.clone() }`. If `Ok(true)`, pick `let backup = sibling(&dst, "backup");`, then call `tokio::fs::copy(dst.path(), backup.path()).await`. If the copy fails, run `cleanup_backups(&completed)` and return `SnapshotFailed`. On success, push `Snapshot::Existed { dst: dst.clone(), backup }`. If `Ok(false)`, push `Snapshot::DidNotExist { dst: dst.clone() }`.
   - **Write:** pick `let tmp = sibling(&dst, "tmp");`, call `tokio::fs::copy(src.path(), tmp.path()).await` to stream the source content to the temp file (no in-memory buffering), then call `tokio::fs::rename(tmp.path(), dst.path()).await` to atomically swap. If either `copy` or `rename` fails, call the internal `rollback` helper described below with `completed` as input, then return either `TransactionalWriteErr::RolledBack { ... }` or `TransactionalWriteErr::RollbackIncomplete { ... }` based on what `rollback` reports. On rename failure also remove the orphan `tmp` (best-effort) before invoking rollback, so we don't leak it.
3. If the loop finishes without error, run `cleanup_backups(&completed)` (best-effort) and return `Ok(())`.

Internal `rollback` helper:

    async fn rollback(snapshots: &[Snapshot]) -> Vec<FileSysErr>

Walks `snapshots` in REVERSE order (most recent first). For each:
- `Existed { dst, backup }` → call `tokio::fs::rename(backup.path(), dst.path()).await`. This atomically restores the original. If it fails, push the error into the accumulator AND leave the backup file in place so the operator can find it.
- `DidNotExist { dst }` → call `tokio::fs::remove_file(dst.path()).await`. If it fails AND the underlying `io::Error::kind()` is not `NotFound`, push the error into the accumulator. (`NotFound` means the rollback raced with another delete and is effectively success.)

Continue through ALL snapshots even if some restores fail — partial rollback is better than aborting rollback. Return the accumulated errors. The caller branches: empty vec → `RolledBack`; non-empty vec → `RollbackIncomplete`. On `RolledBack`, the caller still cleans up any remaining backups (the rollback already consumed the `Existed` ones via rename); on `RollbackIncomplete`, backups are intentionally retained.

Internal `cleanup_backups` helper:

    async fn cleanup_backups(snapshots: &[Snapshot])

Iterates over `snapshots` and removes each `Existed { backup, .. }` file via `tokio::fs::remove_file`, ignoring `NotFound` errors. Logs other failures at `tracing::warn!` level but never returns an error — this is best-effort housekeeping.

This helper does NOT remove parent directories that may have been created during the destination write. Empty directories are left on disk per D3.

Logic:

1. Create a per-call temp directory via `filesys::Dir::create_temp_dir("miru-deploy")` (or whichever helper the codebase uses for ephemeral directories under the agent's working tree). Bind it to a local variable so its `Drop` impl removes the directory at end-of-scope. If the helper does NOT return a self-deleting type, capture an explicit cleanup step at the end of the function and on every error path (best-effort `temp_dir.delete().await.ok()`).
2. Initialize `let mut writes: Vec<(filesys::File, filesys::File)> = Vec::with_capacity(cfg_insts.len());`.
3. For each `cfg_inst` in `cfg_insts`, in order:
   - Build `let dest = filesys::File::new(PathBuf::from(&cfg_inst.filepath));`
   - Call `validate_filepath(&dest)?;` (fails fast on the first malformed input — this short-circuits before any disk I/O).
   - Read the cached content: `let content = content_stor.read(cfg_inst.id.clone()).await?;` (the existing `read` method on the unchanged `CfgInstContent = DirCache<CfgInstID, String>` type alias, which returns a `String`).
   - Pick a temp filename: `let src = filesys::File::new(temp_dir.path().join(&cfg_inst.id));` (the cfg_inst id is unique within a call, so collisions inside the per-call temp dir are impossible).
   - Write the content atomically to the temp file: `src.write_string(&content, WriteOptions::OVERWRITE_ATOMIC).await?;` (using whatever the existing atomic-write call on `File` is named in this codebase).
   - Drop the in-memory `content` (let it go out of scope before the next iteration). Memory cost across the loop is one config's bytes at a time, exactly the same memory profile as the second-pass design.
   - Push `(src, dest)` onto `writes`.
4. Call `filesys::transactional::write_all(&writes).await`. Translate errors via `translate_transactional_err` (defined below).
5. Cleanup: if the per-call temp directory is RAII (self-deleting on `Drop`), nothing to do — the variable goes out of scope at function return. Otherwise call `temp_dir.delete().await.ok()` (best-effort) before returning. The cleanup MUST run on BOTH the success path and every error path; if the helper isn't RAII, structure the function with a `let result = ...; let _ = temp_dir.delete().await; result` tail or an equivalent guard so the path through `?` operators still cleans up.

The `translate_transactional_err` helper is unchanged from prior revisions. The verbatim EACCES/EROFS reason strings are unchanged. The transactional helper's `(File, File)` contract is unchanged.

## 11. Idempotence and Recovery

- Source-code edits are reversible via `git checkout` on the affected files.
- The `chgrp` / `chmod` / `install -d` commands in the manual check are idempotent. Recovery from accidentally broken DAC is the inverse `chown` / `chmod`.
- The lockdown drop-in install in the manual check is reversible: `sudo rm /etc/systemd/system/miru.service.d/lockdown.conf && sudo systemctl daemon-reload && sudo systemctl restart miru.service`.
- The systemd unit edit is reversible via `git checkout agent/build/debian/miru.service` followed by repackage / reinstall and `sudo systemctl daemon-reload && sudo systemctl restart miru.service`.
- After the fifth-pass revision (D13), `write_files` snapshots each destination to a sibling backup at `<parent>/miru.backup.<name>` (no UUID suffix — the deploy state machine guarantees only one apply task runs at a time, so there's no risk of concurrent calls colliding on backup file names). On success, `cleanup_backups` removes each `Existed` snapshot's backup. On failure, `rollback` walks the snapshots in reverse and either renames the backup back over the destination (`Existed`) or deletes the destination (`DidNotExist`). Rollback failures are logged at `tracing::error!` level and never propagated; the deploy state machine sees the original error from the failing iteration. If a previous apply task crashed mid-deploy and left a `miru.backup.<name>` file on disk, the next apply's `snapshot()` call passes `Overwrite::Allow` to `dst.copy_to(&backup, ...)` so the leftover backup is silently overwritten with a fresh snapshot of the current destination — no collision, no error. Re-running a deployment is idempotent regardless of how the previous attempt ended; the next deploy attempt overwrites whatever the previous one left behind.
- No destructive migrations. No database changes. No state file rewrites.
